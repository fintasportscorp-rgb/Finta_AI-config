01 · Agent identity

The bootstrap contract: who the agent is, what it may touch, and what it must never touch. Scope is locked at bootstrap — the agent cannot widen it mid-run.

Lowercase, hyphenated, < 64 chars.
The architecture stays model-agnostic either way.
If success can’t be measured, the loop can’t know when to stop — the forge will flag it.

02 · Loop profile

Three questions pick one of five loop profiles. Start simple; add autonomy only when the proof of success is reliable.

Is the task recurring?
Is the result objectively measurable?
Does it depend on time or an external system?
Is the process stable and well defined?

Answer the questions above — or pick a profile directly.

03 · Guardrails

Emergency brakes, enforced in the runner — not documented in prose. The loop must always be guaranteed to stop.

Hard cap on automatic loop turns before HTL.
N rounds without score progress → stop and escalate.
Above this uncertainty, the agent must CLARIFY, not guess.
One diagnosis, one atomic fix per attempt; then BLOCKED.

04 · Human-to-loop triggers

When the loop hands control back to a human. All five are recommended — disable one only if you know why.

05 · Verification criteria

“Goal verified?” is never one boolean. Deterministic criteria run a command and read the exit code. Soft criteria compare against a spec you validate once — after that the agent applies it autonomously.

Score ≥ threshold → RESOLVED. Below → new iteration. At the iteration cap → HTL.

06 · Orchestration

Strong coupling → one agent. Independence → fan-out. Dependence → sequential. Mixed → both. If decomposition benefit ≤ coordination cost, stay single.

Sub-agents

They coordinate through memory/exchange.md — a JSON-lines bus every agent reads and writes. A failing sub-agent logs to memory_temp.md and re-enters the loop (capped), then HTL.
